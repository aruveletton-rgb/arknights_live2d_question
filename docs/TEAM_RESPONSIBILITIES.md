# 人员职责

当前计划采用四人、三线程。以下内容是职责说明，不代表对应成果已经完成。

## 1. 线程 A1，1 号：桌宠客户端与 Live2D

负责用户在本地电脑上看到和操作的体验：

- 调查并确认 Electron 桌宠模式的启动方式。
- 实现透明、无边框、置顶、拖动和托盘操作。
- 加载 Live2D，控制表情、动作、音频播放和口型同步。
- 提供文字输入、语音入口、设置、错误提示和重连。
- 连接 3 号提供的 VM-1 后端。
- 编写面向新人的客户端启动、配置、模型切换和故障说明。

计划完成标志：桌宠能启动、加载合法占位模型、使用 Mock 独立演示，并能处理后端断线。

## 2. 线程 A2，2 号：角色包与合规

负责桌宠是谁、怎样说话和使用什么素材：

- 设计角色标识、显示名、模型名和 Prompt 组合方式。
- 编写 persona、语气、情绪、动作、直播和版权边界规则。
- 准备知识说明和角色包替换方法。
- 与 1 号确认模型支持的表情和动作。
- 与 3 号确认后端只返回双方能识别的表现名称。
- 登记模型、图片、声音、背景和字体的来源与许可。

计划完成标志：角色设定可被 3 号加载，表现名称与 1 号一致，所有素材有可复核授权说明。

## 3. 线程 B，3 号：后端编排与 LLM

负责接收客户端输入并组织统一回复：

- 调查上游后端入口、WebSocket、角色加载和服务配置。
- 提供健康检查和统一对话入口。
- 管理会话，加载 2 号角色设定并拼接 Prompt。
- 调用 LLM 或 Mock LLM，生成文字、表情和动作。
- 调用 4 号的 TTS 服务，把音频信息返回给 1 号。
- 在 LLM、角色或会话服务失败时返回清楚的降级结果。

计划完成标志：无真实 LLM Key 时仍可使用 Mock 完成单轮对话，输出能被 1 号客户端理解。

## 4. 线程 C，4 号：语音与服务器部署

负责语音能力和远端运行环境：

- 提供 TTS 和 ASR 服务或轻量 API 代理。
- 处理音频缓存、过期清理和服务失败。
- 负责四台服务器的部署、端口、自动重启和低内存配置。
- 提供健康检查、日志限制、配置备份和故障排查方法。
- VM-2 失败时切换 VM-4；语音全部失败时保留文字回复。

计划完成标志：语音服务可独立检查，部署步骤可重复，故障能定位到具体服务器。

## 5. 跨线程决定

| 决定 | 主导 | 必须参与确认的人 |
| --- | --- | --- |
| 表情和动作名称 | 2 号 | 1 号、3 号 |
| 客户端与后端交换内容 | 3 号 | 1 号、2 号、4 号 |
| 语音服务输入输出 | 4 号 | 3 号、1 号 |
| VM-1 部署方式 | 4 号 | 3 号 |
| 素材与声音公开使用 | 2 号 | 全员和仓库拥有者 |
| 最终完整演示 | 全员 | 仓库拥有者确认 |

任何跨线程决定都应在本说明仓库通过 PR 留下简明结论；实际代码改动仍在对应实现仓库完成。

## 6. 已确认的解释

1 号和 2 号人员计划书仅用于再次确认线程 A 两人的分工：1 号负责 A1 客户端，2 号负责 A2 角色包。它们不会取消 3 号的线程 B，也不会取消 4 号的线程 C。
