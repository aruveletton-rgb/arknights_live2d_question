# 项目基线

本文件说明“我们正在做什么、目前做到哪里、哪些事情能改、怎样算完成”。如果后续计划发生变化，应先更新本文件，再开始大范围工作。

## 1. 项目目标

制作一个可在 Windows 本地运行的 Live2D 桌宠：

1. 用户可以用文字或语音与桌宠交流。
2. 后端根据角色设定生成回复。
3. 桌宠能够显示字幕、播放语音，并切换表情和动作。
4. 后端计划部署到四台轻量服务器，并在部分服务故障时继续提供降级体验。
5. 非技术参与者能够按仓库文档完成分工、提问、提交和验收。

## 2. 当前状态

- 阶段：Phase 0，仓库初始化与共同约定整理。
- 已确认：GitHub 仓库、四人三线程职责、目标流程、PR 作为统一提交入口。
- 尚未验证：客户端能否启动、Live2D 模型能否加载、任何后端接口、语音服务、四台服务器部署。
- 尚未确定：最终代码许可证、正式素材和音色授权、上游项目的具体引入方式、服务器实际连接信息。

因此，在代码和测试记录出现以前，不得把本项目描述为“已经可运行”。

## 3. 工作位置与远程仓库

- 正式远程仓库：<https://github.com/aruveletton-rgb/arknights_live2d>
- 正式分支：`main`
- 所有参与者的改动目标：通过 Pull Request 提交到 `main`
- 本机或服务器上的临时位置不属于项目交付物，重要说明和验证证据必须回到仓库。

## 4. 当前人员模型

当前使用四人、三线程方案：

- 线程 A1，1 号：桌宠客户端、Live2D 表现、客户端通信和运行文档。
- 线程 A2，2 号：角色包、Prompt、知识文件、表情动作规则和素材合规。
- 线程 B，3 号：后端编排、WebSocket/API、LLM、会话和角色加载。
- 线程 C，4 号：TTS/ASR、Docker、四服务器部署、健康检查和备份。
- 全员共同：公共接口、字段枚举、错误处理、演示脚本和最终验收。

1 号和 2 号人员计划书是对线程 A 内部两项职责的再次确认，不得据此撤销线程 B、线程 C 或把 3 号、4 号职责转交给 2 号。

## 5. 允许修改的范围

围绕当前任务，可修改：

- 客户端、角色包、后端、部署脚本和对应测试。
- 项目说明、接口合同、合规记录和审计记录。
- 为上述内容必需的配置模板。

以下内容不能擅自修改或提交：

- 用户或服务器上的业务数据。
- 密码、令牌、私钥、真实 `.env` 文件或带凭据的日志。
- 未授权的官方素材、游戏拆包资源、官方角色语音或声优克隆音色。
- 与当前任务无关的系统配置、服务或仓库历史。

## 6. 共同约定与受控文件

下列内容会跨线程产生影响。修改前必须在 PR 中明确说明，并由四位负责人确认：

- [`docs/API_CONTRACT.md`](API_CONTRACT.md)
- `character_pack/characters/*.yaml` 的字段结构
- `emotion` 和 `motion` 可选值
- `.env.example`

## 8. 远程环境假设

计划资源为四台 `Standard B2ats v2` 轻量服务器，每台约 `2 vCPU / 1 GiB`。这是原计划中的配置描述，本次仓库设置没有登录服务器核实。

基于该假设：

- 不在服务器上运行大型 LLM、重型 TTS 或重型 ASR。
- 优先采用 API 转发、缓存、轻量编排和健康检查。
- 真实地址、账号、密钥不得写入仓库。

## 9. 交付物

最终至少需要：

1. 可运行和可打包的 Windows 桌宠客户端。
2. 可调用的 `/health` 和 `/api/chat` 后端服务。
3. 可替换的角色包和素材来源登记。
4. 可执行的四服务器部署说明和配置模板。
5. Mock、自测、联调、降级和完整演示结果。
6. 面向新人的使用、部署、故障排查和合规文档。
7. A1、A2、B、C 四项工作记录的 Markdown 与 JSON 审计文件，以及最终集成报告。
