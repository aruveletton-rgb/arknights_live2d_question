# 项目流程与阶段计划

本文件把项目拆成可以检查的阶段。一个阶段只有在“完成证据”齐全后才能标记完成。

## 1. 每项工作的统一流程

无论是代码、文档还是提问，都按下面的顺序处理：

1. **说明目标**：写清要解决的问题、负责人和完成标准。
2. **确认边界**：指出会改哪些文件，不会碰哪些内容。
3. **建立分支**：从最新 `main` 建立自己的工作分支。
4. **完成工作**：只修改与本任务有关的内容。
5. **自行检查**：按任务标准实际操作，保存命令和结果。
6. **写工作记录**：工作类 PR 同时提交 Markdown 和 JSON 报告。
7. **发起 PR**：目标分支必须是 `main`，使用模板说明结果和风险。
8. **负责人审阅**：仓库拥有者可以提问、要求修改、接受或拒绝。
9. **合并后确认**：只在 PR 合并后，才把任务标记为正式完成。

提问型 PR 不需要测试或审计报告，只要把问题、已尝试内容和希望得到的决定写清楚。

## 2. Phase 0：仓库初始化与约定冻结

目标：所有人从同一套说明开始，不再依赖聊天记录猜测职责。

| 项目 | 负责人 | 交付物 | 完成证据 |
| --- | --- | --- | --- |
| 项目入口和基线 | 仓库拥有者 | README、项目基线 | 新人可从首页找到全部规则 |
| 人员职责 | 全员，3 号主导总接口边界 | 人员职责文档 | A1、A2、B、C 没有重复负责或无人负责的核心模块 |
| 接口初版 | 四人共同 | `API_CONTRACT.md` | 请求、响应、枚举和失败行为明确 |
| 提交方式 | 仓库拥有者 | 协作流程、PR 指南和模板 | 无代码经验者可在网页提交提问 PR |
| 合规入口 | 2 号主导，全员遵守 | 合规规则、素材来源表 | 每项素材都能记录来源和许可 |

完成门槛：上述文件合并到 `main`，四人阅读并在 PR 中确认自己的职责和跨线程约定。

## 3. Phase 1：三线程并行做出最小演示

### 线程 A1，1 号：桌宠客户端

- Windows 桌面出现透明、无边框、可拖动的桌宠窗口。
- 可以加载合法授权的占位 Live2D 模型。
- 可以用 Mock 数据显示回复、播放音频并切换一种表情和动作。
- 服务不可用时显示提示，不导致客户端崩溃。

完成证据：启动步骤、短演示记录、Mock 测试结果、已知问题清单。

### 线程 A2，2 号：角色包

- 建立完整的 `character_pack` 目录和角色 YAML。
- 提供可拼接的 persona、语气、情绪、动作与合规 Prompt。
- 表情和动作列表与 1 号客户端一致。
- 所有占位或正式素材都进入来源登记表，未授权内容不进入仓库。

完成证据：目录和字段检查、Prompt 拼接检查、素材来源表、角色包替换说明。

### 线程 B，3 号：后端编排

- 后端可本地启动，`/api/health` 返回正常状态。
- `/api/chat` 接收文字并返回符合约定的固定 Mock 响应。
- 能加载 2 号角色包；LLM 尚未接入时明确使用 Mock。
- 能调用 4 号的 TTS Gateway Mock，失败时仍返回文字。

完成证据：启动步骤、示例请求与响应、角色加载测试、接口测试脚本和结果。

### 线程 C，4 号：语音与部署

- `/api/tts` 可使用 Mock 或轻量提供方返回可播放音频。
- `/api/asr` 可使用 Mock 或 API 代理返回文字。
- VM-2、VM-3、VM-4 的 Docker Compose 初稿可重复启动或明确标注未验证。
- 健康检查能指出具体服务状态，日志不包含密钥。

完成证据：TTS/ASR 示例、Compose 配置检查、健康检查结果和资源风险说明。

Phase 1 完成门槛：A1、A2、B、C 都有独立自测和对应审计记录，三条线程的最小结果可以按接口合同组合。

## 4. Phase 2：第一次端到端联调

联调顺序：

1. 四人共同核对 `/api/chat`、`/api/tts`、`/api/asr` 和 `emotion`、`motion` 枚举。
2. 3 号先接入 4 号的 TTS 服务或 Mock。
3. 3 号再接入 2 号的角色配置和 Prompt。
4. 1 号最后接入 3 号的远端 `/api/chat`，先验证文字，再验证音频。
5. 四人共同验证表情、动作、断线和服务降级。

一次完整成功演示必须出现：

```text
用户输入一句话
→ 后端生成或模拟回复
→ 客户端显示文字
→ 客户端播放语音
→ Live2D 切换约定的表情和动作
```

完成证据：三条线程使用同一版接口合同、端到端测试记录、失败场景记录和未完成项。

## 5. Phase 3：真实服务与四服务器部署

计划部署顺序：

1. VM-1：Gateway + Orchestrator，由 3 号提供服务，4 号负责部署接入。
2. VM-2：TTS Primary，由 4 号负责。
3. VM-3：ASR Gateway，由 4 号负责。
4. VM-4：Fallback + Backup + Healthcheck，由 4 号负责。

以上是总计划确定的目标拓扑；实际可运行性仍需 4 号在每台 `2 vCPU / 1 GiB` 服务器上验证。在验证前不得写成已部署。

必须验证的失败场景：

- LLM 失败：返回明确的固定文字，不让客户端崩溃。
- TTS 失败：仍返回文字，音频字段为空。
- ASR 失败：提示改用文字输入。
- 网络断开：客户端提示并可重试。
- 未知表情或动作：回到安全默认值。

完成证据：每台服务器的部署步骤、健康检查、资源使用、失败切换结果和回滚说明。不得在报告中记录服务器凭据。

## 6. Phase 4：打包与最终验收

最终验收由四人共同完成，仓库拥有者确认：

- [ ] Windows 客户端可按 README 启动。
- [ ] 桌宠可拖动、置顶、透明显示。
- [ ] Live2D 模型可加载，缺失时有清楚提示。
- [ ] 文字输入能够获得回复。
- [ ] 语音输入能够获得回复，失败时可改用文字。
- [ ] TTS 音频能播放，失败时仍显示文字。
- [ ] `emotion` 和 `motion` 能驱动客户端表现。
- [ ] 网络断开后可提示并重试。
- [ ] 四服务器方案有实际部署或明确的未验证说明。
- [ ] 新人可以只看文档完成基本启动和操作。
- [ ] 素材、声音、代码许可证和来源记录完整。
- [ ] A1、A2、B、C 和最终集成都有 Markdown、JSON 两种审计报告。

## 7. 每日同步

建议每天两次，每次只回答固定问题。

上午：

1. 昨天完成了什么？
2. 今天准备做什么？
3. 是否被其他任务阻塞？
4. 是否准备修改共同约定？

晚上：

1. 今天的 PR 或工作分支是什么？
2. 做了哪些实际检查，结果是什么？
3. 还有哪些未完成或有风险？
4. 明天第一件事是什么？

“没有测试”也是允许的答案，但必须写明原因，不能写成“已通过”。
