# 阶段计划

本文件说明计划顺序。实际开发应在单独的实现仓库中完成，本说明仓库只记录阶段定义、公开状态和需要确认的问题。

## Phase 0：说明和共同决定

- 明确四人三线程职责。
- 明确系统流程、表现名称、服务边界和目标服务器拓扑。
- 确认合规边界和提问方式。
- 建立未来实现仓库的公开链接。

完成标志：四人阅读并确认职责；仍有争议的内容记录为待回答问题。

## Phase 1：三线程分别完成最小演示

- A1：桌宠使用 Mock 显示文字、播放音频和切换表现。
- A2：角色设定可以加载，Prompt 可以组合，素材来源可以核对。
- B：后端使用 Mock 接收文字并返回统一结果。
- C：TTS、ASR 和部署配置可以独立检查或明确使用 Mock。

本说明仓库只记录“对应实现仓库链接、负责人声明和公开状态”，不接收这些代码或产物。

## Phase 2：第一次集成

计划顺序：

1. 3 号接入 4 号的 TTS 服务。
2. 3 号接入 2 号的角色设定。
3. 1 号接入 3 号的对话服务。
4. 四人验证文字、音频、表情、动作和失败降级。

完成标志：有一轮可重复演示，并由负责人提供实现仓库中的可复核证据链接。

## Phase 3：四服务器验证

- VM-1：Gateway + Orchestrator。
- VM-2：TTS Primary。
- VM-3：ASR Gateway。
- VM-4：Fallback + Backup + Healthcheck。

4 号负责实际部署验证，3 号配合 VM-1。说明仓库只记录结论，不保存地址、凭据、部署配置或日志。

## Phase 4：最终验收

- Windows 桌宠能启动和操作。
- 文字与语音输入能获得回复。
- Live2D 能根据约定切换表现。
- 服务失败时有清楚提示和降级。
- 四服务器方案经过实际验证或明确标记未验证。
- 素材、声音和许可证说明完整。
- 非技术人员能按说明理解项目并提出问题。

“代码已写”不等于“阶段完成”；必须有实际验证证据，且证据保存在实现仓库或安全的执行环境中。
