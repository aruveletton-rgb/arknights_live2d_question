# 资料来源与整理决定

本文件说明仓库规则来自哪些材料，以及遇到冲突时采用了哪一种方案。原始文件未复制进仓库；仓库保存的是便于执行的整理结果。

## 1. 使用的材料

- `Arknights VTuber Project Plan.pdf`：27 页总体目标、四人三线程方案、阶段、风险、验收和审计要求。
- `GitHub_Collaboration_Workflow_Guide.pdf`：Fork、分支和 Pull Request 到 `main` 的流程。
- `1号人员计划书.txt`：线程 A 内 1 号职责的补充确认材料。
- `2号人员计划书.txt`：线程 A 内 2 号职责的补充确认材料。

## 2. 人员方案

用户已明确：两份人员计划只用于再次确认线程 A 内 1 号、2 号的分工，不是把整个项目改为两人执行。

当前决定：四人、三线程总计划始终有效。1 号负责 A1 桌宠客户端，2 号负责 A2 角色包，3 号负责线程 B 后端编排，4 号负责线程 C 语音与部署。补充计划中的表述如果跨出线程 A，不得据此覆盖 3 号或 4 号职责。

## 3. 分支方案

差异：总体计划使用 `develop` 和多条长期线程分支；GitHub 协作指南要求所有 PR 最终指向 `main`。

当前决定：仓库规模和参与者经验有限，采用最简单的“短期任务分支 → PR 到 `main`”。不建立长期 `develop` 分支，减少同步和选错目标分支的风险。

## 4. 接口字段

材料中出现过 `text` 与 `reply_text` 两种响应字段，以及两套表情和动作名称。

当前决定：补充计划只细化线程 A，不能单独覆盖全项目公共接口。因此 v0.1 使用四人总计划中的 `text` 字段，以及角色包定义的 `neutral/smile/...`、`idle/greeting/...` 枚举，记录在 [`API_CONTRACT.md`](API_CONTRACT.md)。任何后续变化必须由四人确认。

## 5. 服务器拓扑

四服务器部署属于线程 C，不属于线程 A 的 2 号职责。

当前决定：按总计划使用 VM-1 Gateway + Orchestrator、VM-2 TTS Primary、VM-3 ASR Gateway、VM-4 Fallback + Backup + Healthcheck。4 号负责部署和资源验证，3 号提供 VM-1 服务；验证前仍不得声称已经部署成功。

## 6. 仍需确认

- `Open-LLM-VTuber` 使用 submodule、源码副本还是外部依赖。
- 项目代码许可证。
- 正式 Live2D、背景和声音的授权。
- 四台服务器的实际可用资源与网络条件。
- 最终技术栈、真实启动命令和测试命令。

这些项目影响实现或发布时，必须通过单独 PR 记录决定。
