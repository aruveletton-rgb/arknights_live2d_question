# 协作流程

本项目的参与者不需要直接操作正式分支。所有内容都通过 Pull Request（PR）让仓库拥有者查看后再决定是否合并。

## 1. 一张图理解流程

```text
领取或提出任务
→ 写清目标和完成标准
→ 在自己的分支工作
→ 自行检查并记录结果
→ 发起 PR 到 main
→ 仓库拥有者审阅
→ 修改或回答问题
→ 合并后才算正式完成
```

## 2. 开始工作前

参与者先做四件事：

1. 阅读 [项目基线](PROJECT_BASELINE.md) 和自己的[人员职责](TEAM_RESPONSIBILITIES.md)。
2. 用一句话说明今天要解决的问题。
3. 写出完成标准，例如“点击发送后能看到 Mock 回复”，不要只写“做客户端”。
4. 确认不会提交密钥、个人信息或未授权素材。

如果让 Codex 协助，可以直接说：

```text
请先阅读仓库 README、docs/PROJECT_BASELINE.md、docs/TEAM_RESPONSIBILITIES.md 和 docs/API_CONTRACT.md。
我是 1 号/2 号/3 号/4 号，本次目标是……
只修改……，不要修改……
完成后请运行相关检查，并把实际结果写入 Markdown 和 JSON 工作报告，再创建 Pull Request 到 main。
```

## 3. 分支规则

`main` 只保存审阅通过的稳定内容。参与者使用自己的工作分支：

```text
person-1/简短任务名
person-2/简短任务名
person-3/简短任务名
person-4/简短任务名
question/简短问题名
docs/简短文档名
fix/简短修复名
```

例子：`person-1/desktop-window`、`person-2/character-pack`、`person-3/chat-api`、`person-4/tts-gateway`、`question/model-location`。

每个分支只处理一件主要事情。问题、客户端大改和后端大改不要混在同一个 PR 中。

## 4. 工作完成后的自查

提交工作类 PR 前，负责人回答：

- 改了哪些文件？
- 用什么方法检查？
- 实际结果是什么？
- 什么没有检查，为什么？
- 是否改变了共同接口或配置？
- 是否包含密钥、个人信息、未授权素材或大文件？
- 如果失败，怎样回到修改前的状态？

如果某项没有运行，写“未运行”和原因。不能用“应该可以”代替测试结果。

## 5. 工作报告

代码或重要文档 PR 应在 `audit/reports/` 下同时保存：

- 一份 Markdown：给人阅读。
- 一份 JSON：方便以后自动汇总。

两份文件的状态、测试和风险必须一致。模板见 [`audit/README.md`](../audit/README.md)。纯提问 PR 不需要工作报告。

## 6. 发起 PR

PR 的目标必须是：

```text
aruveletton-rgb/arknights_live2d : main
```

PR 作者填写仓库模板。仓库拥有者检查内容、证据、接口影响和合规风险，可以：

- 留言提问；
- 选择“Request changes”要求修改；
- 选择“Approve”表示认可；
- 在确认后合并；
- 说明原因后关闭不采用的 PR。

没有被合并的 PR 仍是建议，不代表正式项目状态。

## 7. 共同约定的修改流程

如果要修改 `API_CONTRACT.md`、字段枚举、角色配置结构、环境变量或端口：

1. 单独发一个小 PR，标题以 `[接口变更]` 开头。
2. 写清旧内容、新内容、原因和受影响的人。
3. 说明现有客户端或后端是否还能继续工作。
4. 1、2、3、4 号都确认后，仓库拥有者再合并。
5. 两边代码分别跟进，不在没有说明时偷偷改变格式。

## 8. 提问流程

不确定职责、文件位置、接口或验收方式时，不要猜。按[零基础 PR 提问指南](PULL_REQUEST_GUIDE.md)编辑 `questions/QUESTIONS.md` 并发起 PR。

仓库拥有者应在 PR 中给出可执行答案。问题解决后，把决定同步到对应正式文档，再合并或关闭提问 PR。
