# 说明仓库协作规则

## 1. 允许通过 PR 提交的内容

- 向仓库拥有者提出一个明确问题。
- 修正职责、流程、术语、链接或事实错误。
- 补充非技术人员需要的解释。
- 提议一项需要书面确认的跨线程决定。
- 更新实现仓库的公开链接和经过核实的状态。

## 2. 不在本仓库提交的内容

- 客户端、后端、TTS、ASR 或部署代码。
- 模型、素材、声音、安装包和构建产物。
- Docker、服务器脚本、配置文件和运行日志。
- 测试数据、审计产物或包含敏感信息的证据。

上述内容应提交到对应实现仓库。本仓库可以保存其公开链接和简短结论。

## 3. 分支名称

只使用两类短期分支：

```text
question/简短问题名
docs/简短说明名
```

所有 PR 的目标分支都是正式仓库的 `main`。不要直接修改 `main`。

## 4. 统一流程

1. 阅读相关说明，确认问题尚未有答案。
2. 在个人副本或新分支中只修改相关说明。
3. 检查文件是 UTF-8，并确认没有上传代码、二进制或敏感信息。
4. 创建指向正式仓库 `main` 的 PR。
5. 仓库拥有者在 PR 中回答、要求修改、接受或拒绝。
6. 需要形成长期规则时，把最终决定写回对应说明文档。
7. 只有合并后的内容才属于本仓库当前规则。

## 5. 跨线程决定

涉及人员职责、公共信息名称、系统边界或服务器分工时：

1. PR 标题以 `[决定]` 开头。
2. 说明旧解释、新解释、原因和受影响人员。
3. 邀请所有受影响人员确认。
4. 仓库拥有者合并后，决定才生效。

本仓库不通过 PR 直接执行相应代码修改；实现人员应在各自实现仓库另行处理。
