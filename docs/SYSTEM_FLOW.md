# 系统流程

本文件用非技术语言说明计划中的数据流，不定义本仓库的代码结构。

## 1. 文字对话

```text
用户输入文字
→ 1 号客户端发送请求
→ 3 号后端加载 2 号角色设定
→ 3 号调用 LLM 或 Mock
→ 3 号调用 4 号 TTS
→ 客户端收到文字、音频、表情和动作
→ 桌宠显示、播放并驱动 Live2D
```

## 2. 语音对话

```text
用户录入语音
→ 4 号 ASR 转为文字
→ 3 号后端组织角色回复
→ 4 号 TTS 生成回复音频
→ 1 号客户端显示文字并播放音频
```

## 3. 计划中的共同信息

| 信息 | 作用 | 负责人关系 |
| --- | --- | --- |
| 会话标识 | 让连续对话属于同一会话 | 3 号管理，1 号传递 |
| 角色标识 | 选择 2 号定义的角色 | 2 号定义，3 号加载 |
| 回复文字 | 显示给用户并交给 TTS | 3 号生成，1 号显示 |
| 表情 | 控制 Live2D 表情 | 2 号定义，3 号返回，1 号执行 |
| 动作 | 控制 Live2D 动作 | 2 号定义，3 号返回，1 号执行 |
| 音频 | 播放角色声音 | 4 号生成，3 号转交，1 号播放 |

具体字段名称仍需四人共同确认。本说明仓库只记录已确认的最终名称，不保存接口实现。

## 4. 失败时的计划行为

- LLM 失败：返回简短的固定文字。
- TTS 失败：仍显示文字，不播放音频。
- ASR 失败：提示用户改用文字输入。
- 网络断开：客户端提示并允许重试。
- 表情或动作无法识别：使用安全的默认表现。
- 服务器资源不足：拒绝新请求并给出清楚错误，不让服务静默崩溃。
