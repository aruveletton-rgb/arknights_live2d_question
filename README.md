# 明日方舟风格 Live2D 桌宠项目

这是一个基于 `Open-LLM-VTuber` 思路制作的非官方同人桌宠项目。目标是在 Windows 电脑上显示可交互的 Live2D 桌宠，并由远端服务提供对话、语音识别和语音合成能力。

> 当前状态：项目仍处于“计划与仓库初始化”阶段。仓库中的功能说明是目标，不代表对应代码已经完成或测试通过。

> 合规提醒：本项目不隶属于《明日方舟》、鹰角网络、Yostar 或任何相关官方实体。公开仓库不得放入未授权的官方素材、游戏提取资源、官方角色语音或声优克隆音色。

## 当前人员安排

当前按四人、三条线程执行。1 号和 2 号人员计划书只用于再次确认线程 A 内两人的细分，不替代线程 B、线程 C 及其负责人：

| 线程 | 人员 | 一句话职责 | 主要结果 |
| --- | --- | --- | --- |
| A1：桌宠客户端 | 1 号 | 做用户能看到、能操作的桌宠客户端 | Windows 桌宠、Live2D 表现、音频播放、客户端接入 |
| A2：角色包 | 2 号 | 做桌宠使用的角色设定和合规资产结构 | 角色 YAML、Prompt、知识文件、表情动作规则、素材登记 |
| B：后端编排 | 3 号 | 做接收输入、组织回复的系统中枢 | `/api/chat`、WebSocket、LLM、会话、角色加载、TTS 调用 |
| C：语音与部署 | 4 号 | 做远端语音服务和四台服务器运行环境 | TTS、ASR、Docker、健康检查、备份和降级 |

详细边界和每阶段交付物见[人员职责](docs/TEAM_RESPONSIBILITIES.md)。

## 项目从计划到完成的五个阶段

1. 仓库初始化：统一文档、职责、接口和提交流程。
2. 三条线程并行完成最小演示：线程 A 完成桌宠和角色包，线程 B 完成后端 Mock，线程 C 完成语音与部署 Mock。
3. 第一次联调：客户端通过 `/api/chat` 获取文字、语音、表情和动作。
4. 部署和降级测试：验证四台轻量服务器方案以及服务失败时的备用行为。
5. 最终验收：打包 Windows 客户端，完成完整演示、文档和审计记录。

每一阶段的“开始条件、交付物、完成证据”见[项目流程与阶段计划](docs/PROJECT_PLAN.md)。

## 怎样向仓库拥有者提问

本项目把 Pull Request（简称 PR）当作一张“请负责人查看的申请单”。提问者只需要在网页上编辑[问题登记页](questions/QUESTIONS.md)，GitHub 会创建个人副本和 PR；原仓库不会立刻被改动。

推荐标题：

```text
[提问] 我不确定角色模型应该放在哪里
```

完整的逐步说明和按钮名称见[零基础 Pull Request 提问指南](docs/PULL_REQUEST_GUIDE.md)。

## 三个最重要的协作规则

1. 不直接改 `main`：所有问题、文档和代码都通过 PR 交给仓库拥有者确认。
2. 不把“计划完成”写成“测试通过”：只有附有实际测试结果时，才可以标记通过。
3. 不提交敏感或未授权内容：密码、令牌、私钥、官方拆包素材和未授权音色都不得进入仓库。

## 常见词语

| 词语 | 简单解释 |
| --- | --- |
| 仓库（Repository） | 项目的网上文件夹 |
| 分支（Branch） | 不影响正式版本的工作副本 |
| Pull Request / PR | 请负责人查看、讨论并决定是否收下某项改动的申请单 |
| `main` | 负责人确认过的稳定版本 |
| Mock | 用固定的假数据临时模拟尚未完成的服务 |
| 验收 | 按清单实际操作，确认结果符合要求 |
